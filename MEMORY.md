**🚨 Trump jawboning 反转定律（永久记忆，2026-04-08）**

**核心原理**：Trump 本质是「交易员」而非「战略家」，他发推力推的资产往往是已经滞涨、需要外力托盘的方向。喊得越凶，越暴露他（或他的盟友）已经建仓或持仓被套，真实意图是找人接盘。反之，他攻击的资产往往是他想做空的廉价筹码。

**三段式判断框架**：
| 阶段 | 表现 | 操作 |
|------|------|------|
| 阶段1 | 试探性喊话，市场反应平淡或反向 | 按原趋势走，可忽略 |
| 阶段2 | 连续2-3天喊话，市场开始怀疑 | 警惕，开始减仓，不追涨 |
| 阶段3 | 连续3天+威胁升级，或实际行政行动落地 | 反转概率≥70%，逆向操作 |

**量化参考**：Trump 连续喊话≥3次 = 反转概率≥70%；实际行政行动落地 = 反转概率≥80%

Claw Code 是一个**自主软件开发**的示范仓库，展示了如何用 AI Agent 系统（而非纯人工）来构建和维护大型项目。

### JSON 数据结构防故障经验（2026-04-17 重要教训）

**问题**：沙僧模块逐条分析新闻标题显示为空。

**根因**：news-data.js 中**同一日期存在两个 sangsha_module**：
1. 第一个（正确的）：包含完整字段 `title`、`full_title`、`韭菜行为总结`、`市场含义` 等
2. 第二个（简化的）：只有 `news`、`买入概率`、`情绪标签`，**没有 title 字段**

JavaScript 解析时，第二个覆盖了第一个，导致 `analysis_results[0].title` 为 undefined。

**排查方法**：
```bash
# 检查每个日期的 sangsha_module 数量
Select-String -Pattern "sangsha_module" -AllMatches
```

**修复方案**：删除重复的第二个 sangsha_module，保留第一个完整版本。

**核心教训**：
- ❌ **禁止同一层级存在重复 key**：JSON 解析时后者会覆盖前者，不会报错但数据丢失
- ✅ **写入前必须验证**：检查关键字段是否存在、数量是否正确
- ✅ **脚本生成数据后必须抽样检查**：不能只看语法是否正确，还要看字段是否完整

**预防措施**：
1. 数据生成脚本必须检查是否已存在相同 key
2. 写入 news-data.js 后运行验证脚本：

   // 验证每个日期的关键模块数量
   ['sangsha_module', 'white_dragon', 'bajie_conclusion'].forEach(key => {

## 沙僧模块数据规范（2026-04-16 确认，永久记忆）

**核心原则**：沙僧分析的是**全量新闻（all_news）**，不是S/A级新闻。

**两条线路必须同时存在**：
| 线路 | 处理对象 | 输出 |
|------|---------|------|
| 🐵 悟空线路 | S/A级优质新闻 | 深度分析 + bayes_analysis |
| 🧅 沙僧线路 | 全量新闻（all_news） | 每条新闻的韭菜直觉分析 |

**沙僧 analysis_results 每条新闻必须包含的字段**：
```javascript
{
  "news": "新闻标题",
  "title": "简称",
  "full_title": "完整标题",
  "summary": "新闻摘要",
  "追高买入概率": 88,    // 0-100
  "抄底买入概率": 3,     // 0-100
  "恐慌卖出概率": 2,     // 0-100
  "观望概率": 10,        // 0-100
  "情绪标签": ["FOMO", "易懂"],  // 可选：FOMO/易懂/恐慌/复杂/抄底冲动/观望/关注
  "韭菜行为总结": "散户疯了！4400只股票上涨，不买就亏了！",
  "市场含义": "顶部风险",  // 可选：顶部风险/分歧加大/恐慌底部/可能有反弹/中性/中性偏多
  "买入概率": 72,        // 综合买入概率
  "is_long_term": false,
  "date": "2026-04-16",
  "decay": 1.0,          // 时间衰减系数
  "weighted_buy_prob": 72
}
```

**教训**：不能只写汇总数据（overall_sentiment + avg_buy_prob），必须对 all_news 中的**每条新闻**都做详细分析。否则沙僧模块失去意义——它的价值在于识别哪条新闻引发了散户追高/恐慌。

### GitHub Pages 缓存问题教训（2026-04-18 永久记忆）

**问题**：Cron job 更新 news-data.js 后，用户报告网站空白，但 git 显示一切正常。

**排查结果**：
- ✅ `git status` = `nothing to commit, working tree clean`
- ✅ `node --check news-data.js` = 语法正确
- ✅ `web_fetch` 远程 news-data.js = 数据完整
- ❌ 但网站仍然显示空白

**根本原因**：**不是代码问题，是 GitHub Pages 缓存延迟**

**解决方式**：
- 等待 1-3 分钟让 GitHub Pages 重新构建
- 用户按 `Ctrl+F5` 强制刷新浏览器

**核心教训**：
1. **不要急于回滚代码**。当 `git status` 显示 clean 且远程数据正确（`web_fetch` 可获取）时，问题很可能是缓存层
2. **先验证远程再决定行动**。用 `web_fetch` 快速确认远程文件状态，比反复 `git push` 更有效
3. **GitHub Pages 有构建延迟**：push 到 GitHub 后不会立即生效
4. **PowerShell Get-Content 可能显示乱码**，但文件本身 UTF-8 编码正确，应用 `[System.IO.File]::ReadAllText()` 验证

**验证远程数据标准流程**：
```powershell
# 1. 先用 web_fetch 验证远程文件
# https://xttey001.github.io/news-website/news-data.js
# https://xttey001.github.io/news-website/index.html

# 2. 如果远程数据正确 → 等待 + 强制刷新浏览器
# 3. 如果远程数据错误 → 才检查本地 + 回滚
```

**故障排查决策树**：
```
网站空白?
├── git status 显示有改动? → 本地有问题，先修复再 push
└── git status 显示 clean?
    ├── web_fetch 远程数据正确? → 等 2 分钟 + Ctrl+F5
    └── web_fetch 远程数据错误? → 从 git 历史回滚
```

│     └── review.html 永远停留在创建时的状态 ✗                 │
